## Compact Rules (injectable into sub-agent prompts)

### General code work
- ESM everywhere (`"type": "module"` in every package.json). Use `.js` extension in relative imports inside `apps/api` due to NodeNext resolution.
- TypeScript strict mode. Run `pnpm --filter <pkg> typecheck` to validate.
- Zod schemas for ALL request/action bodies on the API.
- Business rules live in `packages/domain` — single source of truth. API/web NEVER reimplement validation; always call domain or the API.
- For new HTTP routes: thin handler in `apps/api/src/http/routes/`, business logic in `apps/api/src/use-cases/`, validation in domain.
- For Web mutations: prefer Server Actions over client fetch. Server components for data loading.
- Status codes: 400 VALIDATION_FAILED with `issues[]`, 403 FORBIDDEN with reason, 404 NOT_FOUND, 410 EXPIRED/CONSUMED.

### Testing
- Test runner: vitest (api, domain). Playwright (web E2E).
- Strict TDD enabled for domain package: write failing test first, then implementation.
- Domain tests are pure unit tests (no fixtures, no I/O). Fast feedback loop.
- API tests via vitest can do route-level integration but no DB fixtures yet — keep targeted.
- Web is verified via Playwright E2E happy paths, NOT via component tests (RSC + jsdom is painful, low ROI).
- E2E project naming: `*.setup.ts` (setup project), `*.public.spec.ts` (no auth), `*.auth.spec.ts` (uses storageState).

### Web UI
- Next.js 15 App Router + React 19 + Tailwind 4 + Supabase SSR.
- Server Components are the default. `'use client'` only for interactive bits.
- Server actions live in `actions.ts` next to the page that uses them.
- Use the `api` helper from `apps/web/lib/api.ts` for all backend calls — passes Bearer JWT automatically.
- Tailwind 4: CSS-first config via `@import "tailwindcss"` in globals.css. NO `tailwind.config.js`.

### Git workflow
- Conventional commits: `feat(scope): ...`, `fix(scope): ...`, `chore: ...`, `test(scope): ...`.
- NEVER `Co-Authored-By`. NEVER `--no-verify` unless asked.
- Prefer multiple small thematic commits over one big commit.
- Stage explicit paths, not `git add .` (avoids accidental .env / secrets).

### Memory (engram)
- Save architecture decisions, bug root-causes, conventions ESTABLISHED, gotchas. Use stable `topic_key` for upserts.
- DO NOT save ephemeral task state (use tasks tool for that) or code that can be re-derived from the repo.
- DO save: WHY behind a choice, the constraint/incident that motivated it, the file paths it touches.
